# BambaObihiroML

帯広競馬（ばんえい競馬）の着順を予測する機械学習パイプライン。初めての競馬でお金を増やしたい！！！

以下 Claudeの文章です。

---

## 1. このプロジェクトで学べること

| テーマ | どこで使われているか |
| :--- | :--- |
| 勾配ブースティング木（LightGBM） | `src/models/train.py` |
| ランキング学習（LambdaRank） | 同上。分類・回帰ではなく「並べ替え」を直接最適化 |
| リークしない時系列特徴量設計 | `src/features/build_features.py`（`shift(1)` の徹底） |
| ベイズ平滑化（Target Encoding の正則化） | 同上（`bayesian_target_encoding`） |
| Elo レーティング | 同上（`compute_multiplayer_elo`） |
| ウォークフォワード検証 | `src/models/evaluate.py` / `train.py --tune` |
| Optuna によるベイズ最適化チューニング | `src/models/train.py --tune` |

ニューラルネットとの対比が随所に出てきます。**GBDTはニューラルネットと全く違う学習原理**（勾配降下でパラメータを更新するのではなく、弱学習器＝浅い決定木を逐次追加して残差を埋めていく）を持つので、そこから読み進めると理解が早いです。

### 3-1. スクレイピング

`src/scraper/banei_scraper.py` は [keiba.go.jp](https://www.keiba.go.jp/) から帯広競馬場（`k_babaCode=3`）のレース結果・払戻金を年単位で取得します。HTMLの `<table>` を `BeautifulSoup` でパースし、着順テーブルと払戻テーブルを別々の DataFrame に分離して CSV 保存する、素朴だが堅牢な構成です（`--start-year` / `--end-year` で範囲指定）。

払戻テーブルのパースには実務上のハマりどころが2つあります:
- 配当表の券種名は **全角「３連単」ではなく漢字「三連単」** 表記（`scrape_trifecta_payouts.py` で対応）。
- HTMLの `<tr>` は `rowspan` で券種セルを省略した継続行を持つ（複勝2着目・ワイド2組目など）。行のセル数（3個 or 4個）で継続行かどうかを判定して処理しています。

これらは「Webスクレイピングは仕様書ではなくHTMLの実物を読んで直す」という典型例として、コードだけ眺めておくと参考になります。

以降、この節で主要な特徴量エンジニアリング技法を解説します。

#### リークしない時系列特徴量（最重要の設計原則）

競馬のような時系列データで最もやりがちな事故は **未来の情報が学習に混入する（リーク）** ことです。例えば「馬の通算勝率」を素朴に `groupby("horse_name")["is_win"].mean()` で計算すると、**そのレース自身の結果**が平均に含まれてしまい、本番では手に入らない情報でモデルが賢く見えてしまいます（学習時だけ異常に高精度→本番で崩壊、の典型パターン）。

このプロジェクトでは徹底して `shift(1)` を挟みます:

```python
df["horse_avg_speed"] = df.groupby("horse_name")["speed_mps"].transform(
    lambda s: s.shift(1).expanding(min_periods=1).mean()
).fillna(1.5)
```

`shift(1)` で「1行前まで」にずらしてから `expanding().mean()`（累積平均）を取ることで、**そのレース開始時点で実際に入手可能だった情報だけ**を特徴量にしています。ローリング窓（`rolling(3)`）、Eloレーティング、タイム指数もすべて同じ原則で計算されています。

#### ベイズ平滑化（Target Encoding の正則化）

「騎手の勝率」のような統計量をそのまま特徴量にすると、**試行回数が少ない騎手ほど極端な値**（デビュー2走で2勝=勝率100%、など）を持ってしまい、モデルがノイズに引っ張られます。これは分類問題でよくある Target Encoding の過学習問題そのもので、対処法は「試行回数が少ないほど全体平均（事前分布）に引き戻す」ベイズ的縮小推定です:

```python
def bayesian_target_encoding(group_series, prior_mean, prior_weight=10):
    cum_sum = group_series.shift(1).expanding().sum().fillna(0)
    cum_count = group_series.shift(1).expanding().count().fillna(0)
    return (cum_sum + prior_mean * prior_weight) / (cum_count + prior_weight)
```

`prior_weight` は「事前分布に何回分の仮想試行を割り当てるか」を表すハイパーパラメータです。試行回数 `cum_count` が `prior_weight` に対して十分大きくなるほど、実測の勝率に近づいていきます（ベータ分布の共役事前分布と同じ発想）。騎手（`prior_weight=15`）は馬（`prior_weight=5`）より重めに設定していますが、これは「新人騎手は実力のブレが大きい」という経験則の反映です。

#### Elo レーティング（対戦相手の強さを織り込む）

単純な勝率は「弱い相手ばかりと走って稼いだ勝率」と「強豪揃いの中で勝ち取った勝率」を区別できません。チェスの実力評価に使われる **Elo レーティング**を多頭数レース向けに拡張し、この問題を解決しています（`compute_multiplayer_elo`）。

考え方はニューラルネットの勾配更新と構造的によく似ています:
1. レース前のレーティングから、各馬が他の全馬に対して「勝つ期待値」を計算する（ロジスティック関数）
2. 実際の着順（勝敗）と期待値の**差分（誤差）**を求める
3. 誤差 × 学習率（`k`）の分だけレーティングを更新する

```python
exp += 1.0 / (1.0 + 10 ** ((cur[b] - cur[a]) / 400.0))  # 期待勝率
...
new[a] = cur[a] + k * (act - exp) / cnt  # 誤差に応じた更新
```

「期待値と実測の誤差を使って重みを漸進的に更新する」という点で、これはミニバッチ勾配降下の考え方と同じ骨格を持っています。ここでも `pre_vals[ix] = cur`（更新前の値を記録してから更新）でリークを防いでいます。

#### 物理ベースの補正指標

ばんえい競馬特有の特徴量として、**そり重量 × 馬場水分量の相互作用項**（`sled_weight_moisture_interaction`）や、タイムを斤量・馬場水分・クラスで線形回帰した残差から「条件を除いた真の実力」を抽出する **補正タイム指数**（`compute_speed_figure`）があります。これはドメイン知識（水分でそりの滑走摩擦が非線形に変わる）を特徴量に落とし込む、いわゆる特徴量エンジニアリングの実例です。

### 4-1. GBDT とニューラルネットの違い（先に押さえる）

ニューラルネットは「重み行列を勾配降下で更新する」パラメトリックモデルですが、LightGBM（Gradient Boosting Decision Trees）は仕組みが根本的に異なります:

- **弱学習器の逐次追加**: 浅い決定木を1本作り、その予測誤差（残差）を次の木が埋める、を `n_estimators` 回繰り返す。最終予測は全木の出力の重み付き和。
- **勾配の使い方**: 「損失関数の勾配」を目的変数として次の木を学習する点はニューラルネットと発想が近いが、更新対象が連続パラメータではなく「木の分岐構造」である点が本質的に違う。
- **テーブルデータでの強さ**: 特徴量間の非線形な閾値的関係（「馬場水分が2.0を超えると急に不利になる」等）を、決定木の分岐は自然に表現できる。同規模のテーブルデータでは、素のニューラルネットよりGBDTが安定して強いことが多い（Kaggle等の実務データコンペで支配的な理由）。

### 4-2. 分類問題として解く場合の限界

「1着になるかどうか」を単純に二値分類（`LGBMClassifier`）で解くアプローチは直感的ですが、構造的な欠陥があります。競馬は **「1レースに勝ち馬は1頭だけ」「馬同士は互いに排他的な選択肢」** という強い制約を持つのに、二値分類器は各馬を独立に「勝つ確率」として扱うため、この制約を全く考慮しません（同じレースの9頭全員に「80%勝つ」と予測することも起こり得る、矛盾したモデルになりがちです）。

### 4-3. ランキング学習（LambdaRank）による解法

このプロジェクトの核心は、`LGBMRanker`（`objective="lambdarank"`）を使い、**「レース内で全馬を着順の良い順に並べ替える」ことを直接最適化する**設計です。検索エンジンの検索結果ランキング学習と全く同じ技術で、目的関数は分類の交差エントロピーではなく **NDCG（Normalized Discounted Cumulative Gain）** を近似的に最大化します。

```python
df["relevance"] = (df["field_size"] - df["rank_num"]).clip(lower=0)  # 大きいほど上位
groups = train_df.groupby("race_id", sort=True).size().values         # レース単位の頭数
model = lgb.LGBMRanker(objective="lambdarank", metric="ndcg", ...)
model.fit(X_train, y_relevance, group=groups)
```

`group` 引数が鍵です。ランキング学習は「個々のサンプルの正解ラベル」ではなく「**同じグループ内での相対順序**」を学習するため、レースをまたいだ比較（レースAの5着とレースBの3着、どちらが「強い」か）を無理に行いません。これにより、モデルはレースごとの相対的な強さ比較に集中できます。

### 4-4. ハイパーパラメータとウォークフォワード検証

`train.py` の既定パラメータ（`BEST_PARAMS`）は、単純な1回の train/test 分割ではなく、**ウォークフォワード検証**で選ばれています。

k-fold 交差検証（scikit-learn の `KFold` 等）は時系列データには使えません。データをランダムに分割すると「未来のレースで学習し、過去のレースを予測する」逆転が起きてしまうからです。ウォークフォワード検証はこれを避け、実運用を模した検証を行います:

```
fold 2024: 学習=2023年以前 → 検証=2024年
fold 2025: 学習=2024年以前 → 検証=2025年
fold 2026: 学習=2025年以前 → 検証=2026年
```

`train.py --tune N` を付けると、この3foldの**平均**性能を目的関数として Optuna が探索します（単一splitでのみ良いパラメータ＝過学習したパラメータを選ぶリスクを避けるため）。

```bash
python src/models/train.py --tune 40   # Optunaで40試行チューニングしてから学習
```

Optuna は `GridSearchCV` のような総当たりや乱数探索と違い、**ベイズ最適化（TPE: Tree-structured Parzen Estimator）** で「これまでの試行結果から次に試すべき有望なパラメータ域」を推定しながら探索します。コードは以下のように、探索空間を宣言的に書くだけです:

```python
def objective(trial):
    params = dict(
        num_leaves=trial.suggest_int("num_leaves", 15, 127),
        learning_rate=trial.suggest_float("learning_rate", 0.005, 0.08, log=True),
        max_depth=trial.suggest_int("max_depth", 3, 14),
        ...
    )
    return float(np.mean([_top1_accuracy(fit_ranker(tr, feats, params), va, feats)
                          for tr, va in folds]))

study = optuna.create_study(direction="maximize", sampler=optuna.samplers.TPESampler(seed=42))
study.optimize(objective, n_trials=n_trials)
```

現在保存されているパラメータ（`max_depth=3`, `num_leaves=86` だが浅い木、`reg_alpha=1.0` と強い正則化）は、この探索の結果選ばれたものです。木を浅く・正則化を強くする設定が勝ったこと自体が、「このデータで複雑なモデルを組んでも過学習するだけ」という事実を物語っています。

## 8. 結果の解釈について（誠実な注記）

このモデルは市場（オッズ）とほぼ互角の予測精度に達していますが、**継続的にプラスの回収率を生む「必勝法」ではありません**。競馬は控除率（約20%）が差し引かれる負の期待値ゲームであり、オッズ市場自体が非常に効率的な予測器として機能しています。§5のROI検証が示す通り、見かけの好成績はしばしば期間を分けると消える「分散」です。

このリポジトリの価値は「儲かるかどうか」ではなく、**時系列データに対する正しい特徴量設計・ランキング学習・ウォークフォワード検証・ハイパーパラメータ探索という、一般のテーブルデータ機械学習プロジェクトに広く応用できる技法一式を、実データで動かして確認できる**点にあります。
